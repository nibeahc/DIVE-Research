users/[이름]/raw/ 안의 자소서 원본 파일을 읽고

## 실행 순서

### 0. 읽을 수 없는 파일 변환 (전처리)

파일 확장자에 따라 아래 방법으로 텍스트를 추출한 뒤 파싱을 진행한다.
Claude가 직접 읽지 못하는 파일은 Bash 도구로 변환한다.

#### HWPX (.hwpx)

HWPX는 ZIP 구조이므로 Python으로 XML에서 텍스트를 추출한다.

```python
import zipfile, re

def extract_hwpx(filepath):
    with zipfile.ZipFile(filepath, 'r') as z:
        xml = z.read('Contents/section0.xml').decode('utf-8')
    texts = re.findall(r'<hp:t[^>]*>([^<]+)</hp:t>', xml)
    return '\n'.join(texts)
```

#### HWP (.hwp, 구버전)

pyhwp 라이브러리의 hwp5txt 명령어를 사용한다.

- 설치 경로: `/Users/chaebin/Library/Python/3.9/bin/hwp5txt`
- 추출된 텍스트가 100자 미만이면 내용이 없거나 암호화된 파일로 간주하고 스킵한다.

```bash

```

#### DOCX (.docx)

python-docx 또는 unzip 후 word/document.xml 파싱으로 추출한다.

```python
import zipfile, re

def extract_docx(filepath):
    with zipfile.ZipFile(filepath, 'r') as z:
        xml = z.read('word/document.xml').decode('utf-8')
    texts = re.findall(r'<w:t[^>]*>([^<]+)</w:t>', xml)
    return '\n'.join(texts)
```

### 1. raw/ 파일 읽기

raw/ 하위에는 두 종류의 폴더가 있다:

- `raw/apply/` : 자소서, 지원서, 이력서, 신청서, 포트폴리오 등 지원/신청을 위해 작성한 문서
- `raw/logs/` : 활동일지, 소감문, 결과보고서, 포토에세이 등 활동 중 작성한 기록 문서

**폴더가 없는 경우 (아직 분류 전):**

- raw/ 루트에 있는 파일들을 파일명 기준으로 아래와 같이 분류한다
  - apply: 자기소개서, 지원서, 신청서, 이력, 포트폴리오, portfolio 포함 파일명
  - logs: 소감문, 활동일지, 결과보고서, 포토에세이, 주간활동 포함 파일명
- 분류 결과를 사용자에게 보여주고 확인받은 후 apply/, logs/ 폴더로 이동한다
- 판단이 어려운 파일은 사용자에게 직접 물어본다

각 폴더의 모든 파일에 대해 0단계 방법으로 텍스트를 추출한다.
파일이 없으면 "raw/ 폴더에 자소서 파일이 없습니다" 출력 후 중단

### 2. writing-style.md 생성

`raw/apply/` 파일만 읽고 아래 항목을 분석해서
users/[이름]/writing-style.md 를 생성한다

**파일 우선순위:** 파일명 또는 내용에서 날짜를 파악하여 가장 최근 서류/자기소개서/지원서의 어투를 더 높은 가중치로 반영한다. 오래된 파일은 참고 수준으로만 사용한다.

- 문장 길이 경향 (짧고 간결 / 길고 서술적)
- 자주 쓰는 문장 시작 방식
- 자주 쓰는 어미 (예: ~했습니다 / ~했어요 / ~했다)
- 자주 쓰는 표현이나 단어
- 전반적인 톤 (감성적 / 논리적 / 경험 중심 등)
- 특징적인 문체 습관

### 3. 경험 목록 추출 (중복 처리 포함)

`raw/apply/` 와 `raw/logs/` 모두에서 등장하는 모든 경험을 추출한다
(어투는 apply만 참고하되, 경험 내용은 양쪽 모두 참고하여 더 풍부하게 구성한다)

#### 중복 판단 기준

아래 중 2개 이상 겹치면 같은 경험으로 판단한다

- 활동명 또는 기관명이 동일하거나 유사
- 시기가 겹침
- 핵심 에피소드(사건)가 동일

#### 중복 처리 방법

같은 경험으로 판단된 경우

- 파일은 하나만 생성한다
- 여러 자소서에 걸쳐 나온 표현 중 가장 구체적이고 풍부한 내용을 선택한다
- STAR 각 항목별로 가장 잘 서술된 버전을 골라 합친다
- "이 경험으로 쓸 수 있는 문항 유형" 은 모든 버전에서 나온 것을 합산한다

#### 중복 판단이 애매한 경우

- 일단 별도 파일로 생성한다
- 파일 하단에 "⚠️ 확인 필요: [파일명]과 유사한 경험일 수 있음" 을 표시한다

### 4. 경험 파일 생성

추출된 경험을 templates/experience-entry.md 형식에 맞춰
users/[이름]/experiences/ 안에 파일로 저장한다

파일명 규칙:

- [시기시작연도]-[활동명요약].md
- 예: 2021-창업동아리팀장.md, 2022-카카오인턴.md

시기를 알 수 없는 경우:

- 본문에서 추론 시도 (대학교 n학년, n년 전 등)
- 추론 불가 시 파일명을 0000-[활동명].md 로 저장하고
  메타의 시기 항목에 "확인 필요" 로 표시

### 5. 관심직무.md 처리

users/[이름]/관심직무.md 를 확인한다

**파일이 있고 내용이 있으면:** 그대로 둔다 (건드리지 않음)

**파일이 없거나 비어있으면:**

- raw/ 및 생성된 경험 파일에서 지원 이력, 직무 언급, 목표 서술 등을 분석해 관심직무를 추론한다
- 추론된 직무 목록과 각 직무별 핵심 키워드를 제안한다
- 사용자에게 확인받은 후 관심직무.md 를 생성한다

**직무별 핵심 키워드 작성 원칙:**

- 키워드는 반드시 해당 사용자의 경험 DB(index.md 키워드별 항목)에 실제로 존재하는 것만 사용한다
- 배경지식이나 직무 특성에서 임의로 키워드를 추가하지 않는다
- 사용자가 직접 입력한 키워드도 그대로 유지한다

### 6. index.md 생성

추출된 모든 경험을 바탕으로
users/[이름]/index.md 를 생성한다

형식:

```text
## 카테고리별
- 리더십: [경험명], [경험명]
- 팀워크: [경험명], [경험명]
- 문제해결: [경험명]
- 성장·실패: [경험명]
- 전문성·직무역량: [경험명]
- 가치관·동기: [경험명]

## 키워드별
- 갈등조율: [경험명]
- 데이터분석: [경험명]
- ...
```
