# 프로젝트 개발 주간 보고서 (Development Report)

## 1. 3월 31일: 프로젝트 핵심 아키텍처 및 파이프라인 설계

본 프로젝트는 다수 사용자의 흩어진 원본 자소서 데이터를 파싱하여 '경험 DB'를 구축하고, 원하는 문항과 글자 수에 맞춰 사용자의 기본 어투로 자소서를 자동 작성해 주는 툴을 목표로 기획되었습니다.

### 1.1 핵심 운영 규칙 및 템플릿 설계

- **프로젝트 전체 운영 규칙 확립 (`CLAUDE.md`)**
  프로젝트의 전체적인 폴더 구조, 명령어 작동 방식, 그리고 절대 금지 사항 등 시스템이 지켜야 할 메인 룰베이스를 정의했습니다.
- **경험 파일 표준화 (`templates/experience-entry.md`)**
  경험 파일의 기본 형식을 STAR 구조로 확립했습니다. 관련 시기, 소속 기관, 역할, 카테고리, 관련 키워드를 메타데이터로 부여하여 파편화된 이력을 체계적으로 관리하도록 템플릿화 시켰습니다.

### 1.2 핵심 로직(Skill) 3종 구현

- **경험 자동 파싱 로직 (`parse-experiences.md`)**
  단순 텍스트 복사가 아니라, 이미 존재하는 경험과 동일한지 여부(확실/애매 판단)를 교차 검증하도록 설계했습니다. 확실한 중복일 경우 제일 풍부한 표현으로 자동 병합하고, 시기가 불명확하거나 중복이 애매한 경우 별도 표시(⚠️)를 주어 리뷰할 수 있도록 구성했습니다.
- **맞춤형 자소서 작성 로직 (`write-coverletter.md`)**
  문항 작성 시 단순 나열이나 개조식(bullet point) 작성을 금지하고 실제 자소서처럼 줄글 문단 형식으로 작성하도록 원칙을 세웠으며, 시스템이 사전에 추출된 사용자의 고유 어투(`writing-style.md`)를 무조건 따르도록 강제했습니다.
- **경험 타임라인 조회 로직 (`query-experiences.md`)**
  파편화된 이력을 한눈에 파악할 수 있도록, 활동명, 기간, 기관명, 역할, 핵심 내용 요약이 담긴 표(Table) 형식으로 매핑한 뒤 시기순으로 정렬하여 출력하도록 구현했습니다.

### 1.3 데이터 구조 설계 관련 주요 결정 사항

- **기간 분류 기준 결정을 연도(Year)로 채택**: 학년 기반 분류는 학교 및 상황마다 범주가 달라 비교에 불리하므로, 데이터베이스 필터링과 시계열 관리에 가장 적절한 '연도' 기준으로 경험의 기준점을 통일했습니다.
- **경험 데이터 분리 보관 (MD 파일)**: 스킬(Skill) 디렉토리는 절차적 용도로 한정하고, 실제 이력 데이터는 지식 레이어인 개별 Markdown 파일 형태로 보관하는 구조로 분리 설계했습니다.

## 2. 4월 5일: 작업 환경 구축 및 초기 시스템 폴더 세팅

이론적 설계가 완료된 후 실제 파이프라인 동작을 위한 개발 환경 세팅 작업이 중점적으로 진행되었습니다.

### 2.1 Claude 환경 통합 및 버전 관리 체계 신설

- **Claude Code 도입**
  추후 진행될 파이프라인 구동 및 자동화 테스트를 위해 CLI 환경에서 동작하는 최신 Claude Code 기반 구동 인프라 환경을 머신에 설치하고 시스템 루트를 셋업했습니다.
- **진행 내역 기록용 보관소 구축 (`record/` 폴더)**
  프로젝트 진행 간 각 시스템 세션이 기존 맥락을 잃지 않고 이어서 작업할 수 있도록, 이전 세션이 기록한 `handoff_[날짜].md` 노트를 `record/` 폴더에 주기적으로 아카이브 하도록 로직을 추가했습니다.

### 2.2 범용 사용자 폴더 구조 적용

- **독립적인 개별 사용자 폴더 구성 (`users/채빈/`)**
  특정 1인에 종속된 툴이 아니라 여러 사용자가 별도로 프로젝트 툴을 이용할 수 있도록 '사용자 폴더 기반 확장형 구조'를 설계했으며, 첫 테스트 대상자로 '채빈'의 작업 공간을 신설했습니다.
- 내부적으로 원본 파일을 넣을 **`raw/`**, 파싱 후 산출물이 저장될 **`experiences/`**, 최종 자소서가 위치할 **`outputs/`** 폴더 생성을 통해 데이터 생애 주기 단계를 명확히 분리했습니다.

## 3. 4월 9일: 파이프라인 고도화 및 맞춤형 기능 업데이트

원시 데이터를 본격적으로 다루기 시작하면서, 기존 설계의 구조적 한계를 보완하고 파싱 효율을 높이는 시스템 단위 개선이 이루어졌습니다.

### 3.1 원시 데이터 세분화 및 맞춤형 직무 관리

- **원시 데이터 서브폴더 뎁스 분리 (`apply/`, `logs/`)**
  단순히 문서를 한 곳에 모아두는 것에서 벗어나 용도에 따라 서브폴더로 분리하여 파싱 효율을 높였습니다. 작성자의 어투 분석은 공식 지원서 형태인 `apply/` 폴더 내용에만 가중치를 부여하고, 경험 데이터 추출 시에는 소감문 및 활동일지가 포함된 `logs/` 문서까지 포괄하여 더 풍부한 팩트 기반 데이터 확보가 가능하도록 설계했습니다.
- **`관심직무.md` 도입을 통한 맞춤형 직무 기반 매핑 설계**
  과거 직무 관련도를 18개의 개별 경험 파일에 정적 필드로 욱여넣어 관리하던 비효율적인 방식을 고쳤습니다. 각 사용자마다 다를 수 있는 희망 직무를 `관심직무.md`로 독립시켜 중앙에서 관리하며, 분류 키워드는 잘못된 환각 매칭 방지를 위해 기존 DB(index)에 실존하는 키워드 풀 안으로만 동작 범위를 한정하였습니다.
- **출력 결과물 타겟팅 저장 규칙 적용 (`outputs/[기업명]_[직무명].md`)**
  하나의 기업과 직무 지원 과정에서 여러 문항(지원동기, 장단점 등)이 분할되어 존재하는 자소서의 특성을 반영하여, 동일한 직무로 요청 시 단일 파일 하위에 계속해서 답변이 누적(append)될 수 있도록 설계 규칙을 변경했습니다.

### 3.2 자동 파싱 지원 포맷 스펙 확장 및 DB 처리

- **다국적/다형식 문서 구문 분석 지원 (`parse-experiences.md`)**
  HWP, HWPX, DOCX, PDF 등 다양한 문서 형식에 대한 자동 텍스트 변환 및 파싱 로직(Step 0)이 탑재되었습니다. 이로 인해 문서 포맷에서 오던 파편화 이슈를 최소화하고 데이터 입력 자동화율을 크게 높였습니다.
- **데이터 필터링 및 채빈 프로필 연동 완료**
  100자 이하의 매우 부실한 문서는 자동 필터링 처리하여 경험 DB 퀄리티를 유지시켰습니다. 또한 채빈의 19개 문서(`apply` 14개, `logs` 5개) 모델링을 완료하여 `profile.md`, `writing-style.md`, `index.md`, 18개의 세부 경험 파일을 성공적으로 론칭했습니다.
- **이력 조회 기준 다각화 (`query-experiences.md`)**
  경험을 효율적으로 호출하기 위해 실무경험, 수상경력, 학술·연구, 대외활동·봉사, 교육·자기계발 등 5분류 시스템을 도입했습니다. 추가로, 사용자 관심직무 파일을 확인한 후 관련된 키워드를 훑어 "직무 기반 관련도 순"으로 이력을 커스텀 정렬 및 조회를 가능하게 만들었습니다.

### 3.3 자소서 자동 작성 엔진 품질 테스트

- **분량 제어 기술 검증**
  BGF리테일 영업관리 직무 등 자소서 작성 로직 적용 테스트를 통해 500자(띄어쓰기 포함 기준) 요청에 단어나 구 차원에서 분량을 정확히 스케일링하여 497자로 응답하는 정밀한 글자 수 분량 조절 결과를 확인했습니다.
- **프롬프팅 룰셋 보완**
  작성 요청 시 지원 희망 직무가 미입력된 경우 관심직무 목록에서 우선 선택하도록 유도하는 방어(fallback) 흐름이 마련되었습니다. 또한 하나의 자소서 문항 내에 너무 많은 경험을 병렬 나열하는 것을 막기 위해 참조 활동을 '기본 1개, 최대 2개'로 엄격히 제한하여 STAR 기반 딥다이브 서술을 하도록 규칙 고도화했습니다.
